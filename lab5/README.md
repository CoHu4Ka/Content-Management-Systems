## Описание работы

Цель работы — закрепить важнейшие практики безопасности WordPress:

- управление ролями и паролями;
- своевременные обновления ядра, тем и плагинов;
- базовый hardening (`wp-config.php`, права доступа, `.htaccess`);
- резервное копирование и восстановление;
- настройка плагина **All In One WP Security & Firewall (AIOS)**.

## Шаг 1 — Подготовка среды

1. Открыть `wp-config.php` в корне WordPress.
2. Добавить сниппет из `snippets/wp-config-hardening.php` (блок «Режим отладки»):

```php
define( 'WP_DEBUG',         true  );
define( 'WP_DEBUG_LOG',     true  );
define( 'WP_DEBUG_DISPLAY', false );
```

>  На продакшен-сервере `WP_DEBUG` должен быть `false`.

---

## Шаг 2 — Управление ролями и паролями

### Создание тестового пользователя

1. Панель администратора → **Пользователи → Добавить нового**.
2. Логин: `test-author`, роль: **Автор**.
3. Пароль: минимум 12 символов, содержит буквы, цифры и спецсимволы (например, `Tr0ub4dor&3`).

### Требования к паролям администраторов

| Критерий | Минимум |
|---|---|
| Длина | 12 символов |
| Прописные буквы | есть |
| Строчные буквы | есть |
| Цифры | есть |
| Спецсимволы | есть (`!@#$%^&*`) |

WordPress автоматически оценивает надёжность пароля при его вводе.

## Шаг 3 — Обновления

### Ручная проверка

**Панель → Обновления** — обновить ядро, темы и плагины до последних версий.

### Автоматические обновления

Добавить в `wp-config.php` (или использовать файл `snippets/auto-updates.php`):

```php
// Автообновление ядра (только патчи)
define( 'WP_AUTO_UPDATE_CORE', 'minor' );
```

Добавить в `functions.php` активной темы:

```php
add_filter( 'auto_update_plugin', '__return_true' );
add_filter( 'auto_update_theme',  '__return_true' );
```

## Шаг 4 — Базовый hardening

### 4.1 Запрет редактирования файлов из админки

Добавить в `wp-config.php`:

```php
define( 'DISALLOW_FILE_EDIT', true );
```

После этого пункты меню **Внешний вид → Редактор** и **Плагины → Редактор** исчезнут из панели.

### 4.2 Права доступа на файлы и папки

| Объект | Права | Команда |
|---|---|---|
| Папки | `755` | `find . -type d -exec chmod 755 {} \;` |
| Файлы | `644` | `find . -type f -exec chmod 644 {} \;` |
| `wp-config.php` | `600` | `chmod 600 wp-config.php` |

Автоматизированная установка прав — скрипт `snippets/fix-permissions.sh`:

```bash
sudo bash fix-permissions.sh /var/www/html/wordpress www-data
```

### 4.3 Защита wp-config.php через .htaccess

Добавить в `.htaccess` (файл `snippets/htaccess-hardening.conf`, блок 1):

```apache
<files wp-config.php>
    order allow,deny
    deny from all
</files>
```

### 4.4 Дополнительные правила .htaccess

Полный набор правил см. в `snippets/htaccess-hardening.conf`:

- запрет листинга директорий (`Options -Indexes`)
- блокировка прямого доступа к XML-RPC
- запрет доступа к `.log`, `.sql`, `.bak` файлам
- HTTP Security Headers (X-Frame-Options, X-Content-Type-Options и др.)

## Шаг 5 — All In One WP Security (AIOS)

### Установка

**Панель → Плагины → Добавить новый** → найти «All In One WP Security & Firewall» → установить → активировать.

### Конфигурация

#### User Login — защита от brute-force

| Параметр | Значение | Обоснование |
|---|---|---|
| Login Lockdown | ✅ включено | основная защита |
| Max Login Attempts | `5` | баланс UX и безопасности |
| Login Retry Time Period | `15 мин` | сброс счётчика попыток |
| Lockout Time | `30 мин` | достаточно для блокировки атаки |
| Force Logout | `24 ч` | ограничивает вечные сессии |

#### User Accounts

- Проверить: нет ли пользователя с логином `admin`.
- Если есть → **AIOS → User Accounts → Rename Admin User** → переименовать в уникальный логин.

#### User Registration

- Если регистрация включена: **Manual Approval** → активировать.
- Предотвращает автоматическое создание спам-аккаунтов.

#### Filesystem Security

- **File Permissions** → запустить сканирование → применить рекомендованные права.
- Не выставлять глобальный write (`777`) — это критическая уязвимость.

#### Firewall

| Опция | Статус |
|---|---|
| Basic Firewall Rules | ✅ |
| Bad Query Strings | ✅ |
| XSS Protection | ✅ |
| Disable Directory Browsing | ✅ |

> Начинать с базового уровня, проверить работу сайта, затем повышать.

#### Brute Force

- **Rename Login Page** → изменить URL с `/wp-login.php` на уникальный, например `/login-usm52`.
- Сохранить новый URL в менеджере паролей (1Password, Bitwarden).

#### Scanner / Malware

- **File Change Detection** → активировать → указать email для уведомлений.
- AIOS будет присылать алерт при изменении любого файла WordPress.

#### Database Backup

1. **AIOS → Database Security → DB Backup** → создать резервную копию.
2. Сохранить `.sql`-файл **вне корневой папки сайта** (например, `/home/user/backups/`).
3. Настроить расписание: раз в сутки или раз в неделю в зависимости от частоты обновлений контента.

#### Notifications

Активировать email-уведомления для:

- блокировка IP при brute-force;
- создание нового аккаунта с правами администратора;
- изменение файлов сайта.

## Шаг 6 — Тест защиты от brute-force

1. Выйти из админки (или открыть приватное окно браузера).
2. Перейти на новый URL входа (например, `/login-usm52`).
3. Ввести неверный пароль **6 раз** подряд для пользователя `test-author`.
4. После 5-й попытки — появляется сообщение о блокировке.
5. Проверить блокировку: **AIOS → Dashboard → Locked IP Addresses**.
6. Разблокировать тестовый IP: нажать **Unlock** напротив заблокированного адреса.

**Ожидаемый результат:** IP заблокирован на 30 минут, в логах AIOS зафиксирована попытка.

## Шаг 7 — Восстановление из резервной копии

### Тест восстановления

1. Удалить тестовую запись и произвольное изображение из медиатеки.
2. Открыть **phpMyAdmin** (или инструмент в панели хостинга).
3. Выбрать базу данных → **Импорт** → загрузить `.sql`-файл резервной копии.
4. Нажать **Вперёд** / **Go**.
5. Проверить, что удалённые запись и изображение восстановились.

> Восстановление медиафайлов требует отдельной резервной копии папки `wp-content/uploads/`.  
> «Полный» бэкап = дамп БД + папка `wp-content` + `wp-config.php`.

## Ответы на контрольные вопросы

### 1. Почему `DISALLOW_FILE_EDIT` и права на `wp-config.php` снижают риск post-exploit?

**Post-exploit** — это действия злоумышленника *после* того, как он уже получил доступ к панели (например, через украденные учётные данные или уязвимость в плагине).

`DISALLOW_FILE_EDIT` убирает редактор кода прямо из интерфейса: атакующий не может вставить PHP-шелл в тему или плагин через браузер, даже обладая правами администратора WordPress.

Права `600` на `wp-config.php` гарантируют, что файл читает только системный пользователь веб-сервера. Если злоумышленник получил доступ к файловой системе от имени другого пользователя (через уязвимость в соседнем приложении на shared-хостинге), он не сможет прочитать строку подключения к БД, секретные ключи и соли.

Вместе эти меры *ограничивают горизонтальное движение*: даже частичная компрометация не сразу даёт полный контроль над сайтом.

---

### 2. Выбранные настройки Login Lockdown / Firewall и их обоснование

| Параметр | Значение | Обоснование |
|---|---|---|
| Max Login Attempts | 5 | Типичный пользователь делает ≤3 ошибок. 5 — запас без блокировки легитимного юзера при случайной ошибке. |
| Retry Time Period | 15 мин | Сброс счётчика через 15 минут: инструменты brute-force не успевают накапливать попытки. |
| Lockout Time | 30 мин | Достаточно для прерывания автоматизированной атаки; не слишком долго, чтобы случайно заблокированный пользователь дождался. |
| Rename Login Page | `/login-usm52` | Убирает стандартный `/wp-login.php` из сканеров и ботов, снижая поверхность атаки. |
| Basic Firewall | включён | Базовый WAF блокирует наиболее распространённые векторы (SQLi в URL, XSS, directory traversal) без ложных срабатываний. |

**Баланс безопасность / UX:** 5 попыток и 30 минут блокировки — это стандарт NIST SP 800-63B. Более агрессивные настройки (2 попытки, 24 часа) создают избыточные проблемы для обычных пользователей при опечатках.

---

### 3. Чем отличаются меры защиты на уровне WordPress (плагин/WAF) от мер на уровне веб-сервера и ОС?

| Уровень | Где работает | Что видит | Примеры |
|---|---|---|---|
| **WordPress / плагин** | PHP, после загрузки WordPress | HTTP-запросы, данные форм, сессии WP | AIOS, Wordfence, nonce, капча |
| **Веб-сервер** | Apache / Nginx, до PHP | HTTP-заголовки, URL, метод запроса | ModSecurity WAF, .htaccess правила, rate-limit в Nginx |
| **ОС / сеть** | Ядро, сетевой стек | IP-пакеты, TCP-соединения | iptables / nftables, fail2ban, SELinux, файловые ACL |

**Ключевое различие:** WordPress-плагины могут быть отключены или обойдены, если сама CMS скомпрометирована (например, через RCE в ядре). Меры на уровне веб-сервера и ОС работают *независимо* от состояния WordPress: fail2ban заблокирует IP ещё до того, как запрос дойдёт до PHP. Поэтому грамотная защита строится эшелонировано — на всех трёх уровнях.

---

### 4. Что должен включать «полный» бэкап WordPress и как проверить восстановление?

**Полный бэкап состоит из трёх компонентов:**

| Компонент | Что включает | Команда/инструмент |
|---|---|---|
| База данных | Все таблицы `wp_*` | `mysqldump -u user -p dbname > backup.sql` |
| `wp-content/` | Темы, плагины, загруженные медиафайлы | `tar -czf wp-content.tar.gz wp-content/` |
| `wp-config.php` | Строка подключения к БД, ключи | скопировать отдельно |

**Как проверить, что восстановление работает реально:**

1. Создать тестовую запись и загрузить изображение.
2. Сделать бэкап.
3. Удалить эту запись и изображение.
4. Восстановить из бэкапа (импорт SQL + распаковка `wp-content`).
5. Убедиться, что запись и изображение снова присутствуют на сайте.

> Бэкап, который не проверен восстановлением, — не бэкап. Рекомендуется тестировать восстановление не реже раза в месяц на отдельной среде.
